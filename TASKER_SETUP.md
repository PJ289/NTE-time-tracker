# Tasker Setup Guide

This guide explains how to configure Tasker on Android so it sends play sessions to the NTE server automatically.

## About Tasker

**Tasker** is a powerful automation app for Android. It lets you create profiles (automation triggers) and tasks (actions to run). For NTE tracking, we use it to:
- Detect when the NTE game opens/closes
- Send play sessions to the server

**Download:** [Tasker on Google Play Store](https://play.google.com/store/apps/details?id=net.dinglisch.android.tasker)

**Note:** Tasker is a paid app (usually around 5-7 USD/EUR), but it's a one-time purchase with no subscription or ads.

## What you need first

Before opening Tasker, make sure you have:

- Tasker installed on your Android phone.
- The server running and reachable from your phone.
- The server address, for example `http://<server-ip>:28183`.
- Admin access to the dashboard so you can create a device.

## Quick import (recommended)

If you want a ready-made profile, import [NTE_use_time.prf.xml](NTE_use_time.prf.xml):

1. Open Tasker and go to the **Profiles** tab.
2. Tap the **home icon** at the bottom-left.
3. Choose **Import Project** and select the XML file.

This import creates the profile and both tasks. It does **not** create the server credentials; you must add those as **global variables** yourself (see [Step 2](#step-2-create-global-variables-required)).

**Recommended order:** create the device in the dashboard (Step 1), create the three global variables (Step 2), then import the profile. You can also import first and create the globals immediately afterward, as long as they exist before you test a play session.

**Important:** If the globals are missing or wrong, uploads will fail, but sessions are still appended to the local queue file. After fixing the variables, **open and close the game** to retry the upload (see [Manual sync](#manual-sync-retry-upload)).

## Step 1: Create the Android device in the dashboard

Tasker authenticates with a device ID and device token. You create both from the dashboard.

1. Open the dashboard in your browser.
2. Go to the **Devices** tab.
3. Paste the admin token and save it.
4. In the **Create Device** panel, fill in:
   - **Name**: any friendly name, for example `Android`.
   - **Type**: `Phone`.
   - **Color**: optional.
   - **Test**: leave off for production.
5. Click **Create**.
6. Copy the token shown after creation.
7. In the device list, copy the device ID shown in the row details.

Important: the device ID is generated by the server. You do not create it manually.

## Step 2: Create global variables (required)

The profile and tasks reference three **global** user variables. Do **not** use project or profile variables for these values: on many Tasker versions they are not expanded reliably in HTTP Request headers and URLs, while globals always are.

### When to create them

| Moment | What to do |
|--------|------------|
| **Before first test** (required) | All three globals must exist with correct values before the exit task uploads data. |
| **Before import** (recommended) | Create the globals first, then import [NTE_use_time.prf.xml](NTE_use_time.prf.xml). Import does not remove existing globals. |
| **Right after import** | Acceptable if you create the globals before opening the game. |
| **After changing server or device** | Edit the globals in the **Variables** tab; no need to re-import the profile. |
| **New phone / Tasker reinstall** | Recreate the globals (they are not stored in the profile XML). |

You need Step 1 finished first: `%NTE_DEVICE_ID` and `%NTE_DEVICE_TOKEN` come from the dashboard.

### Variable names and values

Create exactly these names (case-sensitive, including the `%` prefix):

| Variable | Example value | Notes |
|----------|---------------|--------|
| `%NTE_SERVER` | `http://192.168.1.10:28183` | Base URL only, no trailing slash. Use your LAN IP or public URL. |
| `%NTE_DEVICE_ID` | `a1b2c3d4-...` | Exact ID from the dashboard device row. |
| `%NTE_DEVICE_TOKEN` | `eyJ...` or similar | Exact token shown once when the device was created. |

Names use capital letters (`NTE`, `SERVER`, etc.) so Tasker treats them as **global** variables. All-lowercase names would be local to a single task and would not work here.

### How to create them in Tasker

1. Open Tasker.
2. Go to the **Variables** tab (bottom bar). If you do not see it, open the main menu and choose **Variables**, or use method B below.
3. Tap **+** (add variable).
4. For each row in the table above:
   - **Name**: e.g. `%NTE_SERVER`
   - **Value**: your real server URL, device ID, or token
5. Save each variable and confirm all three appear in the list.

**Method B (any Tasker version):** run a one-off task with three **Variable Set** actions (Category: Variables), one per name, then delete the task if you like. Use the same names and values as in the table.

**Check:** open **NTE Playtime Save**, tap the **HTTP Request** action, and confirm the URL field shows your server (not the literal text `%NTE_SERVER`). If it still shows `%NTE_SERVER`, the global is missing or misspelled.

### Security and updates

- Keep `%NTE_DEVICE_TOKEN` private; anyone with it can upload sessions as your device.
- If you change servers later, only update `%NTE_SERVER`.
- Project export/import does **not** backup global values; export Tasker data separately if you migrate phones.

## Step 3: Create the profile

Tasker profiles decide when the automation runs. For this setup, use an application profile.

1. In Tasker, go to the **Profiles** tab.
2. Tap **+** to add a new profile.
3. Choose **Application** as the profile type.
4. Select the NTE game app.
5. Confirm the profile.

Tasker will now ask you to attach a task to this profile. Create the entry task next.

## Step 4: Create the entry task

This task runs when the game opens. Its job is to store the start time.

1. Create a new task and name it something like **NTE Playtime Start**.
2. Add the first action:
   - **Category**: Variables
   - **Action**: Variable Set
   - **Name**: `%GameStartMS`
   - **To**: `%TIMEMS`
3. Add the second action if you want a human-readable time (optional):
   - **Category**: Variables
   - **Action**: Variable Set
   - **Name**: `%GameStartTime`
   - **To**: `%TIME`

You can keep both variables if you want a human-readable time and a raw timestamp.

## Step 5: Create the exit task

This task runs when the game closes. Its job is to calculate the session and send it to the server.

1. Create another task and name it something like **NTE Playtime Save**.
2. Add a Variable Set action if you want to keep the duration locally (optional):
   - **Name**: `%GameDuration`
   - **To**: `(%TIMEMS - %GameStartMS) / 60000`
3. Add a **Write File** action (Category: File) to store the session in a queue file:
   - **File**: `Tasker/Data/nte_queue.csv`
   - **Text**: `%GameStartMS,%TIMEMS`
   - **Append**: ON
4. Add an **HTTP Request** action from category **Net** (next step).

## Step 6: Configure the HTTP Request action

Use these settings in Tasker:

- **Method**: `POST`
- **URL**: `%NTE_SERVER/api/sessions/queue`
- **Content Type**: `text/plain`

### Headers

Add these headers (one per line):

- `Content-Type: text/plain`
- `x-device-id: %NTE_DEVICE_ID`
- `x-device-token: %NTE_DEVICE_TOKEN`

### File

Set the **File to send** (or **File**) field to:

```
Tasker/Data/nte_queue.csv
```

Leave the body empty. Tasker will upload the queue file as the request payload.

### How queue sync works

- Every session is appended to `nte_queue.csv` when the game closes.
- The HTTP request uploads the entire queue file to the server.
- The server checks the last saved session for this device and ignores older entries.
- On success, the queue file is cleared by the task (so only new sessions remain).

If you are building the task manually, add a **Write File** action after a successful upload:
- **File**: `Tasker/Data/nte_queue.csv`
- **Text**: ` ` (single space)
- **Append**: OFF (overwrite)

Tasker warns if Text is empty; using a single space clears the queue and avoids the warning.

### Manual sync (retry upload)

There is no separate sync task or profile in the imported project. To upload pending sessions on Android (for example after fixing `%NTE_SERVER` or credentials, or when a previous upload failed but data is still in `nte_queue.csv`):

1. Open the NTE game briefly so the profile runs the entry task.
2. Close the game so the exit task runs.

On close, **NTE Playtime Save** appends the current open/close interval to the queue file and then uploads the **entire** queue to the server. Sessions that were already in the file are sent again; the server ignores duplicates and entries older than the last saved session for that device (`filtered` in the JSON response).

Opening the game alone does **not** upload anything; you must close it to trigger the HTTP request.

On **Windows**, use **`sync.bat`** instead (documented in [README.md](README.md)); the PC tracker does not use open/close of the game for that.

## Other settings

- You can leave the timeout and retry settings at their defaults.
- If you are using a test server with a self-signed certificate, you may need to enable "Trust any certificate".
- Enable "Continue Task After Error" if you want the task to complete even if the request fails. This is optional but can be useful for debugging.

## Step 7: Save and test

1. Save the profile and both tasks.
2. Open the NTE app on the phone so the profile triggers.
3. Close the app so the exit task runs.
4. Open the dashboard and confirm that the session appears.

## Step 8 (Optional): Add response validation and notifications

By default, the HTTP request runs silently. You can add actions after it to check the server response and show feedback.

### Capture the response

In Tasker, the HTTP Request action automatically sets these variables:

- `%http_response_code`: The HTTP status code (e.g., `200` for success, `401` for auth error).
- `%http_data`: The JSON response body.
- `%http_headers()`: Response headers.

### Add a conditional after the HTTP Request

The structure is:

```
If (%http_response_code != 200)
  Notify: Error
Else
  Notify: Success
End If
```

Follow these steps:

1. Add an **If** action (category Task) after the HTTP Request:
   - **Condition**: `%http_response_code` **is not** `200`

2. Add a **Notify** action inside the If (category Alert):
   - **Title**: `NTE Tracker Error`
   - **Text**: `Upload error code: %http_response_code`

3. (Optional) Add an **Else** action (category Task):
   - Add a **Notify** action inside the Else:
   - **Title**: `NTE Tracker`
   - **Text**: `NTE Game session uploaded successfully`

4. **Important**: Add an **End If** action (category Task) **after everything** to close the conditional block.

The order must be: **If → Notify (error) → Else → Notify (success) → End If**

### Parse the response for more details

If you want to extract `inserted`/`merged` counts from the response, you can do so with a JSON parse action, but for simplicity, just checking `%http_response_code == 200` is usually enough.

For reference, Tasker's variables from HTTP Request include:

- `%http_response_length`: Size of the response in bytes.
- `%http_cookies`: Cookies from the response (if any).

### Notification options

- **Alert on errors only**: use the If action checking for `!= 200`.
- **Show on success only**: add the Else branch with success notification.
- **Always alert**: add notifications in both If and Else branches.

## Notes on server responses

The server returns JSON with these fields:

- `inserted`: Number of new sessions saved.
- `merged`: Number of sessions merged with nearby ones.
- `skipped`: Number of sessions that were invalid or too short.
- `filtered`: Number of sessions ignored because they were older than the last server record for this device.

Example success response:
```json
{"inserted": 1, "merged": 0, "skipped": 0, "filtered": 2}
```

Example error response (auth failure):
```json
{"error": "Invalid device token"}
```

### Common HTTP codes

- **200**: Success; session was saved or merged.
- **400**: Bad request (invalid JSON or payload).
- **401**: Unauthorized (wrong device ID or token).
- **409**: Conflict (duplicate session).

## What to check if something fails

- If you get `401 Unauthorized`, open the **Variables** tab and verify `%NTE_DEVICE_ID` and `%NTE_DEVICE_TOKEN` match the dashboard exactly (globals, not project variables).
- If the HTTP URL or headers still show `%NTE_...` literally, the corresponding global variable was never created or the name is misspelled.
- If nothing appears in the dashboard, confirm `%NTE_SERVER` is reachable from the phone (browser or ping on the same Wi‑Fi).
- If the profile does not trigger, make sure you selected the correct application in the profile.
- If the request fails, verify the JSON body matches the example exactly.

## Notes for production

- Use a real server URL, not a temporary address.
- Keep the device token private.
- If you replace the server later, only `%NTE_SERVER` should change.
